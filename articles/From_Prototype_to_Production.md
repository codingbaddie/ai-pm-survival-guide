# 🌉 從 Prototype 到 Production：為什麼 AI 寫好的 Code，工程師還要開發兩週？

*(這是一篇寫給所有崩潰 PM 的「防身術」，也適合轉發給那些以為「AI 已經寫好 Code 了」的主管看)*

## 「這不是都跑得動嗎？為什麼不能上線？」

身為 PM，你最近一定遇過這個情境：
你用 Claude/ChatGPT 做出了一個超精美的網頁，按鈕會動、圖表會跑，甚至連 RWD 手機版都調好了。
你興沖沖地拿給工程師看：「你看，AI 把 Code 都寫好了，你們貼上去應該很快吧？明天能上線嗎？」

然後你看見工程師原本發亮的眼神死掉了。
接著他告訴你：「這個要修兩週。」

這時候你心裡一定在想：「兩週？AI 兩分鐘就寫出來了，你在唬我？」

---

## 🦄 歡迎來到「原型幻覺 (The Prototype Illusion)」

我想告訴你一個殘酷的真相：**AI 給你的 Code，是在「真空無菌室」裡長大的寶寶。**
它完美、乾淨，但只要一碰到你們公司那堆累績了三年的「Legacy Code (祖傳程式碼)」，如果不穿防護衣，馬上就會死掉。

這就是所謂的 **「整合債 (Integration Debt)」**。

### 1. 它是「孤兒」，沒有父母 (Database Context)
你的 Prototype 裡的圖表數據，是 AI 幫你 `const data = [...]` 捏造出來的假資料。
但真實世界裡，這些數據散落在 SQL DB、Redis、甚至是那台 20 年前沒人敢動的老 Server 裡。
工程師要做的，不是「畫出圖表」，而是**「挖通水管」** —— 把那些髒亂、格式不一的真實數據，接到這個漂亮的圖表上。這工程往往比畫圖難十倍。

### 2. 它不知道「家規」 (Design System & Global Styles)
AI 寫的 CSS 把它自己顧得很好。但一貼進你們的專案：
*   它的 Class Name 可能跟公司原本的衝突，導致首頁按鈕突然變形。
*   它用的顏色是 `blue-500`，但公司規範是 `#0F4C81`。
工程師得一行一行去「消毒」，確保這個外來種不會破壞原本的生態系。

### 3. 它沒買保險 (Error Handling & Security)
AI 的 Code 通常假設「使用者是用戶，且輸入都正確」。
但真實世界的使用者會輸入 Emoji 當帳號、會斷網、會暴力攻擊 API。
那些 AI 沒寫的 `try-catch`、Input Validation、資安防護，才是真正讓系統不會在半夜掛掉的關鍵。而這些，Prototype 裡通常都沒有。

## 🛡️ PM 該如何自救？

下次當你拿著 AI Prototype 去找工程師時，試試換個說法，你的專業度會瞬間飆升：

> 「這是我用 AI 跑出來的**視覺邏輯驗證 (Visual/Logic POC)**。
> 我知道這是在真空環境寫的，裡面的 Data Binding 和 Error Handling 都還沒做。
> 想請你們評估一下，如果要把它因為**整合進現有的架構**，大概需要處理哪些 Technical Debt？」

這句話傳達了三個訊息：
1.  **我懂行**：我知道 Prototype 不等於 Production Code。
2.  **我尊重**：我知道你們的價值在於「整合」與「穩健性」。
3.  **我實際**：我不是來逼死線的，我是來討論落地可行性的。

## 結語

AI 是加速器，不是魔法棒。
它能幫我們省下「從 0 到 60 分」的時間，但「從 60 到 100 分」的那段天堂路，依然需要人類工程師的專業來鋪平。

*如果你也是那個夾在老闆期待與工程現實中間的 PM，轉發這篇文，或許能救你一命。*
